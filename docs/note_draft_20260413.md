開発ログ #011 ｜ 2026.04.13
転職11日目。「動く」と「使える」は、まったく別の話だった。
-
今日のテーマは、一言で言うとこれだ。
画面の中で動いていたアプリを、手の中で動かす。
-
実機で触った瞬間、全部壊れた
Expo Go でスマホに繋ぐ。それだけのはずだった。
Node.js のバージョン違い。ネットワーク接続エラー。バンドルが0%で固まる。
PCの画面では完璧に動いていたアプリが、スマホでは何ひとつ動かない。
3時間かけて、ようやくQRコードからアプリが立ち上がった。
── そして、次の問題が待っていた。
-
縦画面で、UIが崩壊した
横画面では1行に収まっていた「おこづかいクエスト」が、
縦画面では「おこづかいク / エスト」と無惨に折れていた。
モーダルはスクロールできない。テキストは枠からはみ出す。ボタンのラベルが途切れる。
「動いている」と「使える」は、まったく別の話だ。
画面幅に応じてフォントサイズを自動調整する仕組みを作り、全画面に適用した。
-
「誰が読むか」で、文字は変わる
親画面のテキストを、ひらがなから漢字に変換した。
「しょうにん」→「承認」。「ほうしゅう」→「報酬」。「くりかえし」→「繰り返し」。
40箇所以上。地味な作業だ。でも、大人が「しょうにん」と読まされる違和感は、確実に離脱を生む。
一方、子ども画面では逆のことをした。
漢字の上にルビ（ふりがな）を振る。「財布（さいふ）」「装備（そうび）」「特別（とくべつ）」。
React Native にはルビの仕組みがない。だから自分で作った。
同じアプリの中で、読み手によって文字の見せ方を変える。
── 教材づくりと同じだ。読み手が変われば、表現が変わる。
-
親に「調整する力」を渡した
特別クエストのオン/オフ設定パネルを実装した。
全体のオン/オフ。難易度ごとのオン/オフ。トグルひとつで、子どもの画面が変わる。
「この子にはまだ★3は早いな」── そう判断できるのは、親だけだ。
アプリが決めるのではなく、親が決める。その設計思想を、UIに落とし込んだ。
-
クエストを作っても、表示されない
最後に、最大の落とし穴があった。
クエストを作成しても、一覧に何も出ない。エラーも出ない。ただ、消える。
エラーハンドリングを追加して、ようやく原因が見えた。
Supabase の Row Level Security ── データベースのセキュリティが、モバイル版の書き込みを拒否していた。
Web版は認証ありでログインするから通る。モバイル版はPINログインだから、認証の仕組みが違う。
同じデータベースを共有していても、入口が違えばルールも変わる。
-
「自分以外」が触れる準備を始めた
セットアップガイドを書いた。UI/UXレビューガイドも書いた。
「自分が動かせる」から「誰でも動かせる」へ。
ここでも、教員時代の経験が効いている。
自分がわかる教材と、初めての先生が使える教材は、まったく別物だ。
引き継ぎ資料を書くとき、いつも同じことを考えていた。
── この人は、自分と同じ前提を持っていない。
-
今日の実感
実機で自分のアプリを触って、はっきりわかった。
画面の向こう側にいるのは、自分ではない。
小さな指でタップする子ども。片手で操作する親。初めて触る検証者。
全員が「迷わない」ためには、動くだけでは足りない。
── 「伝わる」必要がある。
